3 Approches Architecturales — Stoffice MVP Commercial

Cliquez sur l'approche qui vous convient le mieux

A

⭐ Monorepo Next.js Full-Stack (Recommandé)

Architecture : Un seul projet Next.js avec App Router

  • Frontend : React + Tailwind + shadcn/ui (dashboard + calculateur)
  • Backend : API Routes Next.js (proxy Dalux sécurisé, CRUD tenants)
  • Auth : NextAuth.js (credentials + JWT)
  • DB : PostgreSQL via Prisma (tenants, clés API chiffrées, cache Dalux)
  • Deploy : Vercel + Vercel Postgres
PRO
Un seul déploiement, DX optimal, Vercel-native, rapide à itérer
CON
Couplage front/back, scalabilité limitée long terme
B

Frontend Next.js + Backend API séparé

Architecture : Deux projets — frontend React et API Express/Fastify

  • Frontend : Next.js (SSR pages + React dashboard)
  • Backend : Express.js sur Vercel Functions ou Railway
  • Auth : JWT custom + middleware
  • DB : PostgreSQL (Supabase ou Neon)
  • Deploy : Vercel (front) + Railway/Render (API)
PRO
Séparation claire, API réutilisable, scale indépendant
CON
Double infra, CORS à gérer, plus complexe pour un MVP
C

Next.js + Supabase (BaaS)

Architecture : Next.js frontend + Supabase comme backend complet

  • Frontend : Next.js + Tailwind
  • Backend : Supabase (auth, DB, storage, edge functions)
  • Auth : Supabase Auth (email/password natif, SSO futur)
  • DB : Supabase PostgreSQL (Row Level Security)
  • Deploy : Vercel (front) + Supabase Cloud
PRO
Auth + DB + RLS gratuit, très rapide à builder, multi-tenant natif
CON
Vendor lock-in Supabase, Edge Functions limitées, moins de contrôle

💡 Ma recommandation : Approche A

Le monorepo Next.js est le sweet spot pour un MVP commercial. Un seul repo, un seul déploiement Vercel, API routes intégrées pour le proxy Dalux sécurisé, NextAuth pour l'auth, et Prisma + Vercel Postgres pour la DB. C'est ce que font 90% des SaaS B2B modernes à ce stade. On pourra toujours extraire le backend plus tard si nécessaire.
